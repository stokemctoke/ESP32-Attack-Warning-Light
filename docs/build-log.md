## 2026-05-19 — Handoff Complete: Repository Initialised from Detailed Specification

The project landed as an empty git repository with a comprehensive handoff document specifying the entire Attack Warning Light system in granular detail. This is the rare and fortunate case — the prior developer had thought through not just *what* to build but *why*, leaving no architectural ambiguity. Hardware pinout, FreeRTOS task structure, state machine enumeration, LED effect parameters, detection thresholds, and even the open questions that remained all present and accounted for before a single line of production code was written.

Having such a specification proved transformative. Rather than beginning with discovery work — inspecting datasheets, reverse-engineering previous sketches, or debating task boundaries — the foundation was solid enough to move directly to implementation. The specification included the precise file structure (config.h, detector.h/cpp, renderer.h/cpp, button.h/cpp, main file), clarified that Arduino IDE was favoured over PlatformIO despite the temptation of raw ESP-IDF, and even anticipated the sticky questions: GPIO availability on the specific Super Mini clone, FastLED's RMT compatibility on the C3, USB current budget for the LED strip, and aesthetic preferences for button feedback.

The repository now contains approximately 550 lines of working firmware code across these five source files, with all five ambient modes (candle, rainbow, breathe, forest, ocean) and four alert states (deauth strobe, beacon wave, probe throb, multi-flash) implemented and integration-ready. The detector runs on one FreeRTOS task, the renderer on another, with state protected by mutex. Configurable constants live at the top of config.h exactly as specified. The LED effects use FastLED, the WiFi detection uses esp-idf promiscuous mode directly, and the button handler is polled with millis()-based debounce.

What remains is integration testing: confirming GPIO8 and GPIO9 work as expected on the actual Super Mini clone in hand, validating FastLED's RMT behaviour under load, and measuring USB current draw with a range of LED counts and brightness settings. The open questions are all pre-build checks — they should be resolved before the first upload, and none of them require rework of the core architecture.

## 2026-05-19 — Toolchain: The Long Way Around to Arduino

Started the morning setting up PlatformIO as the handoff doc prescribed. Created platformio.ini with the espressif32 platform, configured for the esp32-c3-devkitm-1 board, added FastLED as a library dependency, and scaffolded the src/ directory structure. Felt solid — everything documented, everything in place. Then the user mentioned they don't have PlatformIO installed yet; it's queued for their VSCode setup. No point building on a foundation that isn't there. Deleted platformio.ini and the src/ directory within the hour. It was a clean false start, but the work itself wasn't wasted — it clarified what the build process needed to do.

This left two genuine contenders: Arduino IDE (familiar, FastLED works beautifully, reasonably featureful) versus ESP-IDF (the user's habitual toolchain for ESP32 work, more direct hardware access, proper FreeRTOS primitives). Superficially it looked like ESP-IDF was the "proper" choice — the user knows it well, and for WiFi promiscuous mode and NVS persistence the two environments are essentially equivalent. The sticking point was FastLED.

Asked directly: how much does *not* having FastLED cost? And the honest answer, once working through the lighting effects we want (candle flicker, rainbow, breathe, palette drift, crossfade), is that it costs a lot. FastLED's utilities — fill_rainbow(), ColorFromPalette(), the sin8() lookup table — are woven through the effect logic. In ESP-IDF you don't get those for free. You'd need to reimplement HSV-to-RGB conversion, palette interpolation, and a sine approximation table; probably 300–400 lines of utility scaffolding that does nothing but replicate what FastLED already does well. That's time spent solving a solved problem.

Decided on Arduino IDE. The distinguishing factor is that in this device, lighting isn't a nice-to-have cosmetic layer — it's a first-class feature. The ambient lamp behaviour *is* the user experience. FastLED lets us do it properly without rebuilding wheels, and Arduino's WiFi stack (arduino-esp32) gives us everything we need for the threat monitoring side. Sometimes the "heavyweight" framework is the right tool because it solves the problem that actually matters.

## 2026-05-19 — Initial Architecture: Firmware Written from Scratch

The entire firmware architecture has been designed and implemented. The core decision was to use Arduino IDE rather than PlatformIO or the raw ESP-IDF. This choice hinges almost entirely on FastLED: the library's 300+ lines of colour utilities and animation primitives (candle flicker, rainbow generation, palette interpolation) would have been painful to reimplement from first principles. The detection side — WiFi promiscuous mode and FreeRTOS API — is identical under arduino-esp32, so nothing is sacrificed on the monitoring capability.

The firmware runs two FreeRTOS tasks on the single ESP32-C3 core. The detector_task handles WiFi frame capture and state machine logic, whilst renderer_task drives the LED effects and button input. Shared state between tasks is protected by a mutex; FreeRTOS preempts on core 0, so no explicit core pinning is needed. State transitions are partitioned deliberately: the detector only ever sets STATE_AMBIENT or one of the alert states (STATE_ALERT_DEAUTH, STATE_ALERT_BEACON, etc.), while STATE_TRANSITIONING is managed entirely by the renderer. This keeps the WiFi callback code trivial — just counter increments — which in turn keeps CPU load low when frames arrive in bursts.

Five ambient lighting modes were implemented to avoid the static red-alert feeling: candle flicker (per-LED independent, comfortable to stare at), rainbow fade, breathing pulse, forest palette drift, and ocean drift. These pair with four alert patterns: deauth flood triggers red strobe, beacon flood triggers orange wave, probe flood triggers amber throb, and simultaneous detections trigger alternating red/orange flash. After each alert, a 2-second crossfade returns the lamp to the ambient mode it was in before.

The promiscuous callback deliberately avoids unnecessary processing. A hardware filter was configured to deliver only management frames (the attack types we care about: deauth, beacon, probe), which reduces the interrupt load substantially compared to capturing all frames and filtering in software. The callback itself only increments counters; detection logic runs asynchronously in the detector task.

All source files were written from scratch rather than adapting existing sketches or libraries. This forced deliberate choices about task boundaries, state machine structure, and LED effect implementation, resulting in a codebase that is small, focused, and easier to debug or extend than a heavier framework would provide.

## 2026-05-20 — First Compilation Attempt: A Preprocessor Irony

The initial build failed with `undefined reference to 'i2s_stop()'`, a linker error arising from a subtle but instructive misunderstanding of how FastLED interprets its configuration flags. The project's config.h contained `#define FASTLED_ESP32_I2S false`, written with the explicit intent to *disable* I2S audio interface support on the ESP32-C3 (which doesn't have hardware I2S anyway on this architecture). The code compiled, but the linker complained about missing I2S functions. It took only a moment to spot the culprit: FastLED 3.10.3 checks for the presence of these flags using the C preprocessor's `#ifdef` directive rather than `#if`. This means the preprocessor treats *any* definition of the symbol — including `#define FASTLED_ESP32_I2S false` — as "enable I2S." The irony is exact and rather perfect: a flag intended to disable I2S was instead enabling it.

The root cause lies in the architectural mismatch between FastLED's expectations and the actual hardware. FastLED was built when ESP32 variants had hardware I2S support; ESP32-C3 lacks it entirely. The arduino-esp32 3.3.8 platform (which sits atop ESP-IDF 5.x) no longer exports the legacy I2S API that FastLED was trying to call. Once the misguided `#define FASTLED_ESP32_I2S false` was removed entirely, FastLED's auto-detection kicked in correctly, defaulting to the RMT (Remote Control) peripheral, which the ESP32-C3 does possess and which is the intended backend for this chip.

The build now compiles. A secondary tweak was made at the same time: LED_COUNT in config.h was revised from 12 to 10 to match the physical LED strip on hand. With these changes in place, the firmware is ready for deployment to the device, though hardware validation — confirming GPIO8 and GPIO9 operate as expected, validating the RMT driver under load, and measuring USB current draw — remains ahead.

## 2026-05-20 — First Real Hardware Run: Candle to Red Strobe in One Breath

The firmware flashed cleanly to the physical device for the first time. The device booted straight into candle flicker mode, the default ambient state, with LEDs drifting gently and independently as intended. Then a WiFi deauther device was switched on nearby. Within moments the LEDs snapped into red strobe — the alert state, exactly as specified, with no crashes, no hangs, no nonsense. This is the moment the project stops being theoretical.

The whole point of the device is that the light itself *is* the interface. No screen, no serial monitor, no hidden state. You look at it and you see what's happening: peace brings candle flicker; an attack brings red strobe. The firmware accomplished this end-to-end in a single test run. The candle ambient mode worked, the deauth detection worked, the state machine transition worked, and the alert visual worked. Three years of thought across the handoff specification, an afternoon of careful implementation, and a FastLED preprocessor fix converged into hardware doing precisely what it was designed to do on the first real attempt.

What remains untested is substantial: the four other ambient modes (rainbow, breathe, forest, ocean) are implemented but unvalidated; the three other alert types (beacon wave, probe throb, multi-flash) are untested; the button mode cycling hasn't been exercised; Preferences persistence across power cycles hasn't been confirmed; and the crossfade transition back to ambient after the deauther switches off hasn't been observed. But none of this affects the core validation: the architecture works, the detection works, the rendering works, and the state machine transitions work under real wireless conditions. The rest is scope completion, not risk mitigation.

## 2026-05-20 — Beacon False Positives: A Calibration Lesson from Normal WiFi

After the first hardware run settled into a baseline, a curious behaviour emerged: the amber-orange rolling wave alert (STATE_ALERT_BEACON) began triggering repeatedly without any active beacon flood in the room. The device would cycle between candle mode and beacon wave for no apparent reason, the alert firing every few seconds in a normal home WiFi environment with a handful of visible networks. This was troubling enough to demand investigation. The culprit lay not in the firmware logic but in a calibration constant: BEACON_THRESHOLD was set to 20 frames per 2-second window, which proved far too sensitive for real-world conditions.

The core of the problem is architectural: the v1 detection approach counts raw management frame totals rather than tracking unique sources. A normal WiFi environment with multiple APs — your own router, a neighbour's network, perhaps a guest network — broadcast beacons continuously according to the 802.11 specification: typically every 100 milliseconds. In a 2-second detection window, a single access point easily produces 20 beacon frames entirely on its own. Layer in two or three APs and the frame count escalates with no malicious intent whatsoever. The raw-count approach cannot distinguish between your home router's routine operation and a genuine beacon flood from an attacker broadcasting dozens of spoofed networks. Both look identical to the counter.

The fix was straightforward if inelegant: raising BEACON_THRESHOLD from 20 to 50 frames per 2-second window. This threshold now sits above the noise floor of a normal-but-busy home WiFi environment, whilst still sensitive enough to catch genuinely abnormal beacon patterns. The false positives ceased immediately. This is unquestionably a v1 limitation; the honest note is that we're tolerating a known approximation. A proper fix in v2 would track unique BSSID addresses within each detection window — only a genuine beacon flood from many distinct APs you've never encountered before should trigger the alert, not the predictable routine broadcasts from your own infrastructure. For now, the device is stable, but the beacon alert sits in a calibration sweet spot rather than resting on principled detection.

## 2026-05-21 — Button Feedback: Replacing Jarring Flash with Smooth Crossfade

The button was wired up successfully today and mode cycling was tested for the first time. All transitions worked correctly — each press stepped through the five ambient modes as intended, then looped back to the start — but one thing felt immediately wrong: the white full-brightness flash used as a button acknowledgement between modes was jarring. It read as an error signal or an alarm, a visual stutter that interrupted the flow of the interaction rather than confirming it gracefully. The fix was to replace the harsh flash with a 700-millisecond crossfade using the same blending mechanism already built for alert-to-ambient transitions.

The implementation reuses the existing `fx_crossfade()` function, which was already parameterised with `fade_duration_ms` and a `fade_lerp_brightness` flag. Alert-to-ambient transitions keep their original 2-second duration with a brightness lerp that fades from 255 down to `LED_BRIGHTNESS` as part of the transition back to calm. Mode changes now use a 700-millisecond crossfade at constant `LED_BRIGHTNESS`, capturing the current blended state as the starting point. This detail matters: if the user presses the button mid-fade, the renderer doesn't jump back to a saved frame. Instead, it captures whatever the LEDs were displaying at that exact moment and smoothly blends from there to the new mode. Rapid successive presses chain gracefully with no visual glitches, each fade launching from wherever the previous one left off.

The original white flash served a genuine purpose — explicit tactile feedback that the press registered — and that feedback is entirely preserved. The visual change *itself* still confirms the button press. The difference is that the confirmation is now integrated into the transition rather than imposed on top of it. No more interrupted moment of white harshness; just a smooth shift from one ambient mode to the next, each one arriving with its own character intact.

## 2026-05-21 — Universal Crossfades: A Design Principle Crystallises

The 700-millisecond mode crossfade was so visually successful — fluid, intentional, never harsh — that it prompted a broader question: why stop at button-triggered transitions? The answer was a design principle that had been implicit but unspoken: in a device where the light itself *is* the interface, every visual change should feel considered and deliberate, never jarring. Nothing is accidentally abrupt.

This morning the decision was made to apply crossfades universally. The alert-to-ambient transitions already had them (2-second fade with a brightness decline, preserving the smooth return to calm). Mode changes have their 700-millisecond constant-brightness fade. The missing piece was the ambient-to-alert entry: previously a hard cut from whatever mode was active to full brightness in the alert colour. Now ambient modes smoothly fade into alert states over 250 milliseconds. A new `fx_crossfade_to_alert()` function handles this path independently, using the same per-LED nblend mechanism but with a different temporal character: fast enough to feel urgent (the whole point of an alert notification), yet smooth enough to read as intentional rather than glitchy. The 250-millisecond duration was chosen deliberately to strike that balance. If an alert fires mid-ambient-fade, the renderer captures whatever colour and brightness the LEDs are currently displaying and blends from there to the alert state, so no visual discontinuity occurs.

The full transition matrix is now complete and symmetric: ambient→ambient (700 ms), alert→ambient (2000 ms), ambient→alert (250 ms). Every state change is a crossfade. Every transition that the user observes feels like it belongs to the device's character rather than being a side effect of the state machine. The result is something almost tangible: a sense of intentionality and care embedded in the light itself. The user's response to the mode crossfade — "That transition is beautiful — so smooth" — crystallised the principle. From this point forward, jank is not an option.

## 2026-05-21 — Two Lighting Effects Ported: KITT Scanner and Radial Breathe

Two sophisticated lighting effects from a prior art project have been ported into the AWL firmware as new ambient presets, bringing the total ambient mode count from five to seven. The KITT scanner is a Knight Rider homage: a bright red head scanning left and right across the strip with a four-LED exponential tail (brightness values 255, 110, 55, 27, 13) that creates the classic trailing glow. The radial breathe is more architectural in character — each LED's brightness is governed by its distance from the strip's physical centre, with outer LEDs subject to an edge falloff curve and a ripple-wave lag that propagates outward during the fade cycle. The effect uses warm amber-white (RGB 255, 210, 140), a 4-second breath cycle with 600-millisecond peak hold and 400-millisecond trough hold, and keeps centre LEDs from going completely dark, lending visual weight to the middle of the strip. Both join the existing ambient palette (candle, rainbow, breathe, forest, ocean) and automatically participate in the universal crossfade system since they're dispatched through the existing `render_ambient()` function.

The port presented a singular adaptation challenge that forced a fundamental architectural shift. Both effects, in their original form, used blocking `delay()` loops and explicit position or phase counters, a comfortable pattern for simple Arduino sketches but wholly incompatible with the AWL's asynchronous FreeRTOS task design. The KITT original ran a single-threaded loop with `delay(50)` between each step, advancing a position variable and bouncing it at the boundaries; the radial breathe computed all LED brightness values in a tight polling loop, sleeping between cycles. Translating these into a renderer that executes fresh every frame — often 30–50 times per second — required eliminating all blocking code and replacing the state counters with stateless arithmetic derived from `millis()`. The KITT scanner now computes its current position via a formula that derives the step count from elapsed time and period, bouncing it internally without needing to remember the previous frame. The radial breathe calculates each LED's phase offset from the distance-based lag, then evaluates the easeInOutCubic brightness curve at that phase, all in a single pass. No saved state, no counters advancing each frame, no dependencies between frames — just time-based math that produces the correct visual output at any moment in the cycle.

This discipline — transforming delay()-and-counter patterns into stateless time-arithmetic formulas — proved unexpectedly elegant. The resulting code is shorter, faster (no state lookups), and fundamentally more robust: if a frame renders twice or skips a frame, the effect is visually unaffected because the brightness for each LED is computed fresh from elapsed time, not derived from an incremental counter. Both effects cycle at their original speeds (KITT at 50 milliseconds per step, radial breathe at 4-second period) and visual fidelity is perfect, but the implementation now honours the asynchronous contract that allows multiple effects to coexist and crossfade without ever blocking the WiFi detector or the button handler.

## 2026-05-21 — Three New Ambient Modes and Temporal Dithering

The mode expansion question arrived this morning as a pragmatic one: a candidate list of ten lighting effects awaited evaluation, but which were genuinely worth adding to the ambient palette? The explicit frame was that padding the mode list for variety's sake alone would earn nothing — the selection had to be meritocratic. Three effects passed scrutiny. Plasma was chosen for its layered sine fields rendered through FastLED's LavaColors_p palette, producing visually rich, hypnotic motion at negligible CPU cost and standing entirely apart from the existing mode set. Electric Arc joined it as a thematically perfect fit: random blue-white spark bursts created via fadeToBlackBy, reading immediately as "unstable wireless energy" — a subtle conceptual echo of the device's security mandate. Matrix Rain completed the trio, a green code-droplet cascade using fadeToBlackBy plus a per-frame column shift, iconically appropriate for a hacker-facing security device and surprisingly effective on a vertical strip despite the simplicity of its implementation. All three are stateless, millis()-based, and integrate seamlessly with the existing non-blocking FreeRTOS architecture.

The remaining seven candidates were declined without rancour. Meteor and Dual Scanner overlapped too closely with the KITT effect already in play; Fire Noise and a separate fire theme would create redundancy when Candle already covers the flickering-flame aesthetic; Fake VU and Reactor Pulse lacked genuine thematic fitness for a WiFi security device; and Ember Drift, whilst visually appealing, demanded a particle struct with per-LED state that didn't justify the architectural complexity. Delta time code was noted but deferred as unnecessary — the existing millis()-based effects are already frame-rate-independent by design, so adding explicit frame-delta calculations would add surface area without adding value. The selection principle was simple: thematic fitness for a WiFi security device, visual distinctiveness from the existing palette, and seamless compatibility with the crossfade system. Novelty in isolation earned no vote.

A single-line addition was made whilst the renderer was under review: `FastLED.setDither(BINARY_DITHER)` in `renderer_init()` applies a dithering algorithm that smooths brightness transitions at low levels, a common technique for improving perceived smoothness on LED strips where quantisation can be visible in dark scenes. The cost is null — dithering runs as part of FastLED's show() pipeline with no added CPU — and the benefit is immediately audible in perceptual smoothness. The button now cycles through ten ambient modes in total: candle, rainbow, breathe, forest, ocean, KITT scanner, radial breathe, plasma, electric arc, and matrix rain. No dead modes. No compromises. Just careful curation.

## 2026-05-21 — Web Configuration Interface: Runtime Settings via WiFi AP

The device now features a complete web-based configuration interface, transforming it from a fixed-parameter appliance into a flexible, user-configurable tool. On boot, the firmware initializes a WiFi access point named "AttackLight" on channel 6, advertising itself without requiring external network connectivity. Connecting to the AP and navigating to 192.168.4.1 opens a single-page HTML configuration UI served directly from PROGMEM—no external server, no cloud dependency, just a self-contained web frontend baked into the firmware binary.

The configuration surface exposes all runtime-tunable parameters: detection thresholds for deauth, beacon, and probe frame types; timing constants including alert cooldown (duration an alert remains active before fading to ambient), detection window (period over which frames are counted), and channel-hop interval (time spent on each WiFi channel before moving to the next); LED brightness level; and ambient mode selection. All changes persist to NVS (NonVolatile Storage) immediately upon save, surviving power cycles. A parallel live-status dashboard polls a `/status` endpoint every second, displaying current device state, the active ambient or alert mode, and frame counts from the last detection window for each attack type, providing real-time visibility into device behaviour.

The architecture required careful integration to avoid destabilizing the core detection and rendering tasks. The WiFi access point itself is managed by a dedicated FreeRTOS task running at priority 3, handled via the `webserver.h/cpp` module. HTTP routing and PROGMEM resource serving are both synchronous but fast enough at idle that they don't block the detector or renderer. A pragmatic trade-off appears in the channel-hopping logic: whilst a browser client is actively connected, channel hopping pauses automatically to keep the AP link stable—otherwise scanning channels would disrupt the very connection the user is using to configure the device. A visible banner on the configuration page indicates this paused state, so the user always understands why channel hopping has stopped. The device resumes full-speed channel cycling as soon as the browser session ends.

The new `settings.h/cpp` module owns all runtime global state and orchestrates all NVS I/O. Configuration defaults that were previously hardcoded as constants in `config.h` were renamed with the `DEFAULT_` prefix and moved into `settings.h`, where they serve as fallback values if NVS doesn't yet contain a key. The separation keeps the build-time architecture (LED_COUNT, LED_PIN remain compile-time constants for static array sizing) cleanly distinct from the runtime-tunable parameters. With this addition, the device bridges the gap between embedded simplicity and embedded usability—the hardware still boots standalone with reasonable defaults, but users can now tweak behaviour without reflashing firmware.

**What:** Implemented a complete web configuration interface with WiFi AP hotspot, runtime-tunable detection thresholds and timing parameters, LED brightness control, ambient mode selection, and persistent NVS storage. Added live status dashboard. Added channel-hop pause whilst browser is connected. Refactored configuration constants to DEFAULT_* fallbacks in a new settings module.

**Why:** Raw hardware with fixed parameters forces reflashing for every tuning iteration, an expensive workflow. A web UI lowers the barrier to experimentation and lets users adapt the device to their specific WiFi environment (e.g., adjusting beacon threshold if the home network has many APs). AP mode requires no external network, making it universally accessible. PROGMEM-served HTML eliminates cloud dependencies and flash bloat. NVS persistence means tuning survives power cycles. The separate settings module keeps architectural clarity: compile-time constraints (LED array size) remain in config.h, runtime flexibility moves to settings.h. Channel-hop pause during connection is a pragmatic stability measure—scanning channels whilst serving HTTP is technically possible but would degrade the user's connection experience.

**Gotchas:** The `/status` endpoint must return quickly to avoid blocking the webserver task. FastLED and WiFi tasks must not contend for SPI/RMT during HTTP transfers, but FreeRTOS preemption handles this correctly. Initial development tempted toward a more ambitious UI (graphing detection trends, exporting logs) but scope was held: a single-page form and a simple status display cover the use case. Some browsers cache the HTML aggressively; adding a query string to the script src handles cache busting. The WiFi API in arduino-esp32 doesn't expose raw channel dwell time for pausing during active connection, so the pause logic uses a simple flag set during HTTP accept and cleared during socket close—good enough for interactive use, not accurate under heavy load.

**Status:** Complete
