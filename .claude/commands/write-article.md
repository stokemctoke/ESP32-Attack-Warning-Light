## Voice and tone

- Friendly and conversational, like explaining something cool to a mate who is also into tech
- Technically accurate and specific, do not dumb things down
- Accessible to a tech and maker savvy audience, not a corporate one
- Occasionally drop in a dry or nerdy joke where it fits naturally, never forced
- UK English throughout
- No stiff formal language, no buzzwords, no "In conclusion" type closings

## Article structure

1. **Opening hook** - What is this project and why does it exist? Make it interesting.
2. **The build journey** - Walk through the major decisions in rough chronological order, grouping related choices. For each: what was decided, why, and what went wrong or sideways.
3. **What I learned** - Honest reflection. What worked well, what was a pain, what would be done differently.
4. **Closing** - Where the project is now, what is next if relevant, invite the reader to ask questions or share their own approach.

## Formatting

- Use H2 and H3 headers to break up sections
- Keep paragraphs short, three to five sentences max
- Use inline code formatting for component names, pin numbers, function names, and library names
- Do not use bullet lists for the main body, write in prose
- Include a suggested title and a one sentence meta description at the top of the file as a comment
